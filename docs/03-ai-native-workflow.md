# AI 原生工作流：从阶段交接到连续证据流

## 本章入口

| 项目 | 内容 |
| --- | --- |
| 本章目标 | 把传统阶段交接流程改造成连续证据流 |
| 适合谁 | 需要改造研发、内容、运营或客户流程的人 |
| 预计耗时 | 18 分钟 |
| 最终产出 | 一个 AI 工作流规格 |

## 旧流水线的问题

传统软件或内容生产常见流程是：

```text
需求 -> 设计 -> 排期 -> 执行 -> 测试 -> 发布 -> 运营
```

这个模式的核心是阶段交接。它适合执行速度较慢、信息传递成本较高的时代。

AI 出现后，前半段产出速度大幅提升：文档、原型、代码、测试用例、营销素材都可以快速生成。如果组织方式不变，后半段的评审、验证、发布和维护会被压垮。

## 新流程：连续证据流

AI 原生流程不是把旧流程加速，而是把任务变成一条连续证据流。

```text
目标 -> 假设 -> 规格 -> 实现 -> 验证 -> 发布 -> 观察 -> 复盘
```

每一步都要留下可以检查的证据。

| 阶段 | 关键问题 | 证据 |
| --- | --- | --- |
| 目标 | 解决什么用户问题 | 用户反馈、业务指标、机会判断 |
| 假设 | 为什么这个方案可能有效 | 假设列表、反例、风险 |
| 规格 | 做到什么程度算完成 | 验收标准、边界、样例 |
| 实现 | 如何执行 | 代码、文档、素材、任务记录 |
| 验证 | 怎么证明它没偏 | 测试、评审、对照样例 |
| 发布 | 怎么安全上线 | 灰度、回滚、权限 |
| 观察 | 上线后看什么 | 指标、日志、用户反馈 |
| 复盘 | 下次如何更好 | 决策记录、模板更新 |

## AI 在每一环的位置

| 阶段 | AI 可以做 | 人必须做 |
| --- | --- | --- |
| 目标 | 汇总反馈、发现模式 | 决定优先级 |
| 假设 | 生成方案、找反例 | 选择假设 |
| 规格 | 起草验收标准 | 确认边界 |
| 实现 | 生成初稿、代码、测试 | 审查关键逻辑 |
| 验证 | 跑检查、找异常 | 判断风险是否可接受 |
| 发布 | 生成发布说明、监控清单 | 批准上线 |
| 复盘 | 整理材料、提取规律 | 决定组织改进 |

## 一个任务规格应该包含什么

最小可用规格：

- 目标：这件事解决什么问题？
- 用户：谁会受影响？
- 非目标：这次明确不做什么？
- 上下文：AI 和团队必须知道哪些背景？
- 验收：什么情况算完成？
- 风险：哪些地方可能出错？
- 发布：如何上线，如何回滚？
- 复盘：完成后沉淀什么？

## 工作流改造的顺序

不要一开始就追求全自动化。建议按以下顺序推进：

1. 标准化输入：先把任务说明写清楚。
2. 标准化验收：再把完成标准写清楚。
3. 标准化复盘：让经验能复用。
4. 半自动执行：让 AI 参与高频环节。
5. 自动化检查：用脚本、测试和监控守住质量。
