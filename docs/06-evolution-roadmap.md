# 进化路线图

## 本章目标

这一章回答：传统个体和传统团队应该按什么顺序进化。

不要一开始就追求自动化。正确顺序是先建立目标、上下文和验证，再谈提速。

## 个体进化路线

### 第 1 阶段：从任务执行到目标理解

先训练自己不直接执行任务，而是先问：

- 真实问题是什么？
- 谁需要这个结果？
- 什么标准说明完成？
- 哪些事情这次不做？

这是从执行者到负责人心智的第一步。

### 第 2 阶段：从零散提问到上下文管理

把常用背景、样例、风格、约束和错误整理成可复用上下文。

目标是让每次 AI 协作不从零开始。

### 第 3 阶段：从单次生成到任务链协作

不要只让 AI 生成最终答案，而是让它参与完整任务链：

```text
理解 -> 拆解 -> 初稿 -> 反向检查 -> 修改 -> 验证 -> 复盘
```

### 第 4 阶段：从主观判断到验证体系

为自己的产出建立验证方式：

- 事实核对。
- 样例对比。
- 反例审查。
- 自动测试。
- 用户反馈。

### 第 5 阶段：从个人技巧到个人系统

当目标、上下文、任务链和验证可以重复使用时，个人就开始形成自己的 AI 操作系统。

这时你不只是会用 AI，而是有一套稳定放大自己的系统。

## 团队进化路线

### 第 1 阶段：统一语言

团队先定义清楚：

- 什么是 AI 可以做的。
- 什么必须由人判断。
- 哪些信息不能给 AI。
- 哪些输出必须被审核。

没有共同语言，后续所有工具都会变成个人偏好。

### 第 2 阶段：选择一个价值流

不要全公司一起改。先选一个真实、频繁、有明确结果的价值流：

- 新功能开发。
- 用户反馈分析。
- 内容生产。
- 客户报告。
- 销售线索跟进。

### 第 3 阶段：把任务变成规格

团队要从“口头需求”转向“任务规格”。

一份规格至少包含：

- 目标。
- 非目标。
- 上下文。
- 验收标准。
- 风险。
- 发布和回滚。

### 第 4 阶段：建立连续证据流

每个任务都要留下过程证据，而不是只看最终结果。

证据包括目标判断、方案假设、执行记录、验证结果、上线观察和复盘结论。

### 第 5 阶段：沉淀团队操作系统

把有效做法沉淀为：

- 团队章程。
- 上下文库。
- 工作流模板。
- 权限规则。
- 检查清单。
- 复盘机制。

## 进化顺序

```text
个人目标能力
-> 个人上下文能力
-> 个人验证能力
-> 团队共同语言
-> 团队任务规格
-> 团队连续证据流
-> 团队操作系统
```

## 不建议的顺序

不要这样做：

- 先买工具，再想场景。
- 先上自动化，再定义责任。
- 先让每个人自由使用，再补团队规则。
- 先追求产量，再补验证。
- 先做全员改造，再找一个价值流试点。

## 本章结论

进化不是一口气变成超级个体或超级团队。

正确路径是：先把一个人的工作链路变清楚，再把一个团队的价值流变清楚，最后才把工具和 AI 嵌入进去。
